# V2.14.0-J - Agent File Tool Routing / Read-only Tool Bridge Report

## ToolRouting

`src/agentFileToolBridge.ts` adds a narrow read-only routing layer for agent/runtime file tool calls.

- Supported routes: `stat`, `read`, `list`, `search`.
- Unsupported routes such as `write`, `delete`, and `rename` return `unsupported_file_tool`.
- The bridge does not import `fs` and does not touch files directly.
- `LLMBridgeView.executeAgentFileToolRoute()` delegates to the bridge with `executeFileToolRequest()` as the only runner.

## PolicyGateIntegration

All real operations continue through the existing executor path.

- Route input is converted to `FileToolExecutionRequest`.
- Execution is delegated to `executeFileToolRequest()`.
- `executeFileToolRequest()` calls `executeFileTool()` with the current file access policy and Working Set refs.
- V2.14-I1 realpath/symlink hardening remains inside `executeFileTool()`.

## PendingFlow

Confirm results are not executed by the routing layer.

- External unauthorized read/stat still returns `confirm`.
- Confirm results carrying `pendingRequest` are enqueued by `LLMBridgeView.executeFileToolRequest()`.
- The existing pending external read approval UI remains the confirmation surface.

## ResultSurface

The bridge returns an internal `AgentFileToolRouteResult`.

- It wraps the executor result without changing `AgentEvent` v0.1.
- `formatAgentFileToolRouteResult()` provides a JSON result string for existing workflow/tool-result style containers if a runtime caller needs display text.
- Image/PDF/binary/unknown read results remain refs-only / Claude Read handoff from the executor.

## NoWriteBoundary

No write path is opened in this phase.

- The route allowlist excludes write/delete/rename/diff-write.
- `agentFileToolBridge` contains no file write/delete APIs.
- CLI backend, SDK backend, Skills runtime, and promptPackage are not changed for write access.

## Tests

Added unit coverage verifies:

- Agent/tool route calls for `stat`, `read`, `list`, and `search`.
- Confirm results do not execute directly and still surface a pending read request.
- Deny results return a clear reason.
- Symlink escape remains protected by executor realpath guard.
- Image refs-only and oversized text bounded behavior remain intact.
- No write/delete/rename route exists.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

The bridge is callable from runtime code, but no external CLI/SDK protocol adapter is added in this phase. Future SDK/CLI tool-call adapters must call `executeAgentFileToolRoute()` or `executeFileToolRequest()` and must not read files directly.

## Recommendation

Freeze J if build/unit/process pass and diffs remain limited to read-only routing, tests, and report. Next work can add runtime-specific adapter plumbing, but all adapters should treat this bridge as the only file tool execution boundary.
