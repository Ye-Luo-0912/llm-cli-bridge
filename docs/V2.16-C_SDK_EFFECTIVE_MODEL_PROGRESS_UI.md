# V2.16-C — SDK Effective Model & Live Progress UI

- 基线：`6ea0332`（V2.16-B）
- 范围：修复 SDK 主路径下模型/推理等级与实际运行不一致的问题，完善 SDK 实时过程展示
- 约束：不 tag/release，不改大架构，不恢复 Prompt Snippets

## RootCause

UI 显示 `gpt-5.4·高`，但 SDK log 显示 `gpt-5.5[1m]`，五者不一致的根因：

1. **buildSdkOptions 未传 model/effortLevel**：`src/sdkBackend.ts` 的 `buildSdkOptions` 只传了 `cwd`/`permissionMode`/`includePartialMessages`，未传 `model` 与 `reasoningEffort`，导致 SDK query 使用其内部默认值（gpt-5.5 + 1m），而非用户在 UI 选择的 gpt-5.4 + high。
2. **model picker 硬编码**：`src/view.ts` 内联硬编码 `MODEL_OPTIONS` / `EFFORT_OPTIONS` 常量作为唯一来源，无法从 SDK runtime catalog 动态读取，且 effort 强制中文化（低/中/高/超高）与 SDK 原始名称（low/medium/high/max）不一致。
3. **SDK 过程只靠 log**：`onWorkflowEvent` 回调只把事件 push 到 `sdkEvents` 数组，运行结束后才由 `appendSdkWorkflow` 折叠展示。长任务期间用户看不到 agent 正在做什么。
4. **Assistant message 重复**：SDK 的 `SDKAssistantMessage`（text block）与 `SDKResultMessage`（result text）可能携带相同文本，两者都映射为 `message` 事件并在 live progress 中各自渲染一行，造成视觉重复。

## EffectiveModel

修复 `src/sdkBackend.ts` 的 `buildSdkOptions`，使其传入用户选择的 model 与 effort：

```typescript
const options: Record<string, unknown> = {
  cwd: task.cwd,
  // V2.16-C: 传入 model 与 effortLevel，使 UI 选择与 SDK query 一致
  model: settings.model || undefined,
  // V2.16-C: SDK query 使用 reasoningEffort 传递推理等级
  ...(settings.effortLevel ? { reasoningEffort: settings.effortLevel } : {}),
  permissionMode: settings.claudePermissionMode ?? "default",
  includePartialMessages: false,
};
```

一致性链路（五者一致）：
- **UI 选择** → `setModelEffort()` 写入 `plugin.settings.model` / `plugin.settings.effortLevel`
- **settings** → `saveSettings()` 持久化
- **SDK query options** → `buildSdkOptions(task, settings)` 读取 `settings.model` / `settings.effortLevel`
- **SDK log** → SDKSystemMessage(init) 回传 `model` 字段，映射为 "SDK session started (model=gpt-5.4, ...)"
- **运行过程展示** → live progress 实时显示 session started 行，含 model=gpt-5.4

## ModelCatalog

新增 `src/runtimeModelCatalog.ts`，替代 view.ts 中的硬编码常量：

```typescript
export interface RuntimeModelCatalog {
  readonly models: ReadonlyArray<ModelCatalogEntry>;
  readonly efforts: ReadonlyArray<EffortCatalogEntry>;
  readonly source: "runtime" | "static";
}
```

- **source="static"**：当前 Claude Agent SDK（@anthropic-ai/claude-agent-sdk@0.3.191）未暴露 model catalog API，`tryReadRuntimeCatalog()` 返回 null，fallback 到静态目录。
- **静态模型**：gpt-5.5 / gpt-5.4 / glm-5.2 / deepseek-v4
- **静态 effort**：low / medium / high / max（原始名称，不中文化）
- **归一化函数**：`normalizeModelValue` / `normalizeEffortValue` 确保非法值 fallback 到默认；`findModelEntry` / `findEffortEntry` 用于 chip label 解析。

架构可扩展：未来 SDK 暴露 catalog API 时，只需实现 `tryReadRuntimeCatalog()` 返回 `source="runtime"` 的目录，UI 自动切换。

## ModelPickerUI

重构 `src/view.ts` 的 `renderModelEffortPicker` 为单列紧凑 popover：

- **布局**：`.llm-bridge-model-effort-popover-single` 单列，上半 `.llm-bridge-model-list`（Model），下半 `.llm-bridge-effort-list`（Effort）。
- **label**：使用 catalog 原始名称（gpt-5.4 / low / medium / high / max），不再强制中文化。
- **选中高亮**：`refreshModelEffortPicker` 为当前 model/effort 的 option 添加 `is-active` class。
- **选择即关闭**：点击 option 调用 `closeModelEffortPopover()` + `setModelEffort()`，立即刷新 chip。
- **chip 显示实际 effective model/effort**：`refreshModelEffortPicker` 用 `findModelEntry` / `findEffortEntry` 解析 label，显示 `gpt-5.4 · high` 格式。

删除的硬编码：`MODEL_OPTIONS` / `EFFORT_OPTIONS` 常量已从 view.ts 移除。

## SDKLiveProgress

新增 `appendLiveSdkEvent(ev: WorkflowEvent)` 方法，在 `onWorkflowEvent` 回调中实时调用：

- **实时渲染**：每个非终态事件立即在当前 assistant message 的 `.llm-bridge-live-progress` 区域追加一行（icon + label + detail），不等运行结束。
- **精简展示**：每行显示事件类型图标 + 标签 + 详情摘要（message 截断 80 字符，tool_result 截断 60，tool_start 显示工具名，file_change 显示路径）。
- **终态清理**：`completed` / `failed` 事件将 live progress 设为 `display:none`（DOM 保留用于审计），运行结束后由 `appendSdkWorkflow` 展示完整 SDK Workflow 折叠区。
- **自动滚动**：每行追加后 `scrollIntoView` 到底部。
- **合并重复 Assistant message**：message 事件到达时，若上一行也是 message 且 detail 文本相同（SDKAssistantMessage text == SDKResultMessage result），跳过不创建新行，实现"合并同一次 SDK 输出"。

覆盖的事件类型：session started（system message）、model/cwd、tool_start、tool_result、file_change、permission、message delta、thinking、error、completed/failed。

## Blockers

- **CLI explicit query 超时（环境问题，非回归）**：V2.16-B runtime smoke Phase 3 CLI explicit query 超时（60s 无输出）。CLI 二进制存在（preflight available=true, version=2.1.196），但 API 调用超时，疑为 rate limit / 认证环境问题。V2.16-C 未修改任何 CLI 代码路径，基线 6ea0332 同样存在此环境依赖。
- **SDK model catalog 动态读取**：当前 SDK 包未暴露 model catalog API，`RuntimeModelCatalog.source` 为 "static"。未来 SDK 支持时需实现 `tryReadRuntimeCatalog()`。

## Recommendation

1. **后续可接入 runtime catalog**：当 Claude Agent SDK 暴露 model/effort 列表 API 时，实现 `tryReadRuntimeCatalog()` 返回 `source="runtime"` 目录，UI 自动从 SDK 读取可用模型，无需改 view.ts。
2. **live progress 可扩展为可折叠**：当前 live progress 在运行中始终展开。若长任务事件过多，可考虑加一个"精简/详细"切换，但当前 truncateText 已控制每行长度，20 个事件以内体验良好。
3. **CLI 环境稳定性**：CLI explicit 超时为环境问题，建议在 CI 中单独标注 CLI smoke 为 "manual required" 或增加重试机制。
