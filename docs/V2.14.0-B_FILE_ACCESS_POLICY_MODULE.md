# V2.14.0-B - Shared File Access Policy Module Report

## PolicyTypes

`src/fileAccessPolicy.ts` defines the shared boundary vocabulary for future Composer Attachments, Working Files, File Tools, and Diff Write surfaces.

- `FileAccessOperation`: `read`, `write`, `delete`, `rename`.
- `FileAccessDecision`: `allow`, `confirm`, `deny`.
- `FileAccessRisk`: `low`, `medium`, `high`.
- `FileAccessReason`: stable machine-readable reasons such as `inside_read_root`, `inside_write_root`, `outside_read_roots`, `outside_write_roots`, `sensitive_path`, `path_traversal`, `invalid_path`, and `rename_target_denied`.
- `FileAccessPolicy`: resolved `vaultPath`, `readRoots`, `writeRoots`, and `sensitivePathMode`.

The module is intentionally pure. It does not call Obsidian APIs, does not read or write files, and does not mutate runtime settings.

## Decisions

Default root behavior:

- `readRoots` defaults to `[vaultPath]`.
- External read access is opt-in through `externalReadRoots`.
- `writeRoots` defaults to `vaultPath`.
- `outputDir` is added to `writeRoots` only when it resolves inside the Vault.
- Vault-external write, delete, and rename remain hard deny.

Decision behavior:

- Read inside a configured read root: `allow`.
- Read outside all configured read roots: `deny`.
- Write/delete/rename inside a write root: `allow`.
- Write/delete/rename outside all write roots: `deny`.
- Sensitive paths default to `deny`.
- Sensitive paths may return `confirm` only when a caller explicitly builds a policy with `sensitivePathMode: "confirm"`; this is reserved for future strong-confirm UI flows.

No current runtime path has been switched to this module in V2.14.0-B. This avoids accidentally changing existing CLI/SDK/action behavior before the Working Files and File Tools surfaces are designed.

## PathSafety

The module implements explicit path normalization and containment helpers:

- Windows drive paths are evaluated through `path.win32` and normalized case-insensitively.
- POSIX paths are evaluated through `path.posix`.
- Relative paths resolve against the normalized Vault path.
- Path traversal segments (`..`) are rejected before read/write decisions.
- Containment uses relative-path checks instead of string prefix checks.
- Output directories are normalized before being considered as write roots.
- An output directory that resolves outside the Vault is not added to `writeRoots`.

Sensitive path detection covers:

- `.env` and `.env.*`.
- Filenames containing token, credential, credentials, secret, or secrets markers.
- `.ssh`.
- Common private-key names and extensions: `id_rsa`, `id_dsa`, `id_ecdsa`, `id_ed25519`, `.pem`, `.key`, `.p12`, `.pfx`.
- `.git/config`.
- `.obsidian`.
- `.llm-bridge` credential surfaces such as `bridge.json`, token, credential, and secret files.

## Tests

`scripts/run-tests.mjs` now includes a V2.14.0-B unit section that bundles `src/fileAccessPolicy.ts` and validates:

- Policy types and exported functions exist.
- Default roots are Vault-only for reads and writes.
- External read roots allow reads but do not allow writes.
- Vault-internal outputDir is normalized and added as a write root.
- Vault-external outputDir is ignored for write roots.
- Vault-internal write is allowed.
- Vault-external write/delete/rename is denied.
- Sensitive paths default to deny.
- Strong-confirm mode returns `confirm` for sensitive paths.
- Windows drive containment and POSIX containment are both covered.
- Relative path traversal is denied.
- `promptPackage`, CLI backend, SDK backend, AgentEvent, and Skills runtime are not wired to this module in this phase.

Required validation for this phase:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

- Existing `actions.ts` still uses its legacy local path guard. That is intentional for V2.14.0-B; replacing it should be a separate compatibility change.
- The module is not yet persisted in settings. Future settings work must add explicit external read-root management and avoid granting write roots outside the Vault.
- `confirm` is policy-level only. No UI confirmation workflow is connected in this phase.
- File content ingestion remains out of scope. Future Composer Attachments and Working Files must pass file references and use bounded, explicit reads instead of blindly appending full external files to prompts.

## Recommendation

Proceed next to V2.14.0-C only after accepting this boundary as the single source of truth.

Recommended next implementation step:

1. Add settings storage for explicit external `readRoots`.
2. Keep `writeRoots` derived and non-editable: `vaultPath` plus Vault-internal `outputDir`.
3. Wire Composer Attachments or Working Files to `evaluateFileAccess` for read decisions only.
4. Defer write/delete/rename integration until File Tools and Diff Write have explicit approval UX.
5. Do not connect this module to `promptPackage` as raw content injection.
