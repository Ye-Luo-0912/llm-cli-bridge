# V2.14.0-K - Runtime Tool Adapter / SDK + CLI File Tool Plumbing Report

## RuntimeAdapter

`src/runtimeFileToolAdapter.ts` adds a shared runtime adapter surface for read-only file tools.

- Adapter kinds: `cli`, `sdk`.
- Supported tool calls: `stat`, `read`, `list`, `search`.
- Input normalization accepts common path keys such as `path`, `file_path`, `filePath`, `directory`, and search `query` / `pattern`.
- `LLMBridgeView.run()` creates the adapter for the selected backend and stores it on `AgentTask`.
- CLI and SDK backend helpers (`executeCliRuntimeFileTool`, `executeSdkRuntimeFileTool`) call `task.runtimeFileToolAdapter.execute()`.

## RouteBoundary

The runtime adapter does not touch files directly.

- No adapter imports or uses `fs`.
- All runtime calls normalize input into an `AgentFileToolRouteRequest`.
- Execution flows through `LLMBridgeView.executeAgentFileToolRoute()` and then `executeFileToolRequest()`.
- V2.14-I/I1 executor policy, bounded reads, and realpath/symlink hardening remain the only filesystem execution path.

## PendingFlow

Confirm decisions remain UI-driven.

- External unauthorized read/stat returns `confirm`.
- `executeFileToolRequest()` enqueues the pending read request into the existing external read approval UI.
- The adapter returns the confirm result surface and does not read the target.

## ResultSurface

The adapter returns an internal `RuntimeFileToolAdapterResult`.

- It preserves status, reason, route result, and formatted JSON output.
- Confirm/deny results set `isError=true` for existing workflow/tool-result style surfaces.
- Refs-only image/PDF/binary/unknown behavior remains executor-owned and returns Claude Read handoff text.
- AgentEvent v0.1 is unchanged.

## NoWriteBoundary

No write capability is added.

- `write`, `delete`, `rename`, and diff-write are not supported adapter tools.
- Unsupported tools return `unsupported_file_tool`.
- CLI/SDK backends do not call file executor directly.
- Skills runtime is unchanged.

## Tests

Added coverage verifies:

- CLI and SDK adapter helpers route `stat`, `read`, `list`, and `search`.
- Adapter module does not import or use `fs`.
- Confirm pending flow remains routed to the existing file tool route.
- Deny flow returns clear reasons.
- Image/PDF refs-only and oversized bounded read behavior remain intact.
- Symlink escape is still delegated to executor realpath guard.
- Write/delete/rename routes are unavailable.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

This phase exposes a safe runtime adapter surface and wires it into tasks. Actual external SDK/CLI protocol-specific tool registration may still need a later adapter hook once the target runtime API shape is finalized. That later hook must call this adapter and must not read files directly.

## Recommendation

Freeze K if build/unit/process pass and backend diffs remain limited to adapter plumbing. Future work can add protocol-specific invocation glue, but this adapter should stay the single read-only file tool boundary.
