# V2.12.1 Patch Trigger Report

> 基于 V2.12.0 Manual Required Smoke 用户回填结果，发现 1 个 blocker，触发 V2.12.1 patch 流程。

## 1. Blocker 详情

### ManualId 13: Skill 重命名后 meta 未迁移

| 项目 | 内容 |
|---|---|
| ManualId | 13 |
| 场景 | 重命名后 pinned/applyCount/lastUsedAt 保留 |
| 预期 | pinned/applyCount/lastUsedAt 迁移到新名（V2.11.1 renameSkillMeta） |
| 实际 | 旧名仍保留 meta，新名 meta=null，迁移未生效 |
| 严重性 | Blocker（V2.11.1 核心修复项在真实 UI 流中失效） |

### 复现步骤

1. 导入一个 skill（如「改写润色」）
2. 对该 skill 执行 pin + apply（生成 pinned=true / applyCount=1 / lastUsedAt）
3. 编辑该 skill 并改名（如「改写润色」→「改写润色V2」）
4. 保存编辑
5. 观察 `.llm-bridge/skills-state.json`：
   - **预期**：旧名 meta 迁移到新名，旧名条目删除
   - **实际**：旧名仍保留 meta，新名 meta=null

## 2. 根因分析

### 代码位置

`src/view.ts` openEditSkillDialog（line 2622-2647）

### 问题代码序列

```typescript
// view.ts:2637-2641
this.skillsState = renameSkillMeta(this.skillsState, skill.name, newName);  // 1. 内存迁移
this.scheduleSkillsStateSave();                                              // 2. 500ms 后异步写盘
await this.refreshSkills();                                                  // 3. 立即从磁盘重载 ← 覆盖内存
```

### 根因

`refreshSkills()`（line 2224）在 line 2227 执行 `this.skillsState = await loadSkillsState(vaultPath)`，从磁盘重新加载 state。但 `scheduleSkillsStateSave()` 的 500ms 防抖定时器尚未触发，磁盘上仍是旧 state（meta 在旧名下）。`refreshSkills` 的磁盘重载覆盖了 line 2637 的内存迁移结果，导致 meta 迁移丢失。

### 时序图

```
T0: renameSkillMeta() → 内存 state.skills = {新名: meta}（旧名已删）
T0: scheduleSkillsStateSave() → 500ms 定时器启动
T0: refreshSkills() → await loadSkillsState(vaultPath)
T0+ε: loadSkillsState 返回磁盘 state（仍为 {旧名: meta}）← 覆盖内存
T500: 定时器触发 → saveSkillsState(vaultPath, this.skillsState) → 写入 {旧名: meta}（已被覆盖的旧 state）
```

**结果**：磁盘和内存都是旧 state，meta 迁移完全丢失。

## 4. 最小修复建议

### 方案 A：先 flush 再 refresh（推荐）

在 `refreshSkills()` 前先 flush 待写的 skillsStateSaveTimer，确保磁盘写入完成后再重载。

```typescript
// view.ts openEditSkillDialog 内
if (newName !== skill.name) {
  this.skillsState = renameSkillMeta(this.skillsState, skill.name, newName);
  // V2.12.1: 先 flush 待写 state，避免 refreshSkills 从磁盘重载覆盖内存迁移
  await this.flushSkillsStateSave();
  this.scheduleSkillsStateSave(); // 仍调度一次确保最终持久化
}
```

新增 `flushSkillsStateSave()` 方法：
```typescript
private async flushSkillsStateSave(): Promise<void> {
  if (this.skillsStateSaveTimer !== null) {
    window.clearTimeout(this.skillsStateSaveTimer);
    this.skillsStateSaveTimer = null;
    try {
      const vaultPath = (this.app.vault.adapter as unknown as { getBasePath: () => string }).getBasePath();
      await saveSkillsState(vaultPath, this.skillsState);
    } catch {
      // 写入失败不阻断主流程
    }
  }
}
```

### 方案 B：refreshSkills 不重载 skillsState（保守）

`refreshSkills` 只重载 skills 列表，不重载 skillsState（state 由调用方管理）。但此方案影响面较大，需审计所有 refreshSkills 调用点。

### 推荐：方案 A

方案 A 修改面最小（仅 openEditSkillDialog + 新增 flushSkillsStateSave），不改变 refreshSkills 语义，不影响其他调用点。
