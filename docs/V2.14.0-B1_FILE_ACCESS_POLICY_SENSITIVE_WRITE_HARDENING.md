# V2.14.0-B1 - File Access Policy Sensitive Write Hardening Report

## ChangedBehavior

`src/fileAccessPolicy.ts` now treats `sensitivePathMode: "confirm"` as a read-only strong-confirm mode.

Before B1, a sensitive path returned `confirm` for any operation when the policy was built with `sensitivePathMode: "confirm"`.

After B1:

- `read` + sensitive path + `sensitivePathMode: "deny"` returns `deny`.
- `read` + sensitive path + `sensitivePathMode: "confirm"` returns `confirm`.
- `write` + sensitive path always returns `deny`.
- `delete` + sensitive path always returns `deny`.
- `rename` with a sensitive source path always returns `deny`.
- `rename` with a sensitive target path always returns `deny`.

## SensitiveReadPolicy

Sensitive reads may be denied or escalated to a future strong-confirm UI flow.

This keeps the V2.14-B design goal intact: external and sensitive reads can be governed by explicit user intent without granting write capability.

## SensitiveWritePolicy

Sensitive writes are hard denied regardless of `sensitivePathMode`.

This applies to write, delete, and rename because all three can mutate or destroy sensitive material. Strong-confirm is not a bypass for mutation.

## Tests

The V2.14.0-B unit test section now includes B1 assertions:

- Read `.env` with confirm mode returns `confirm`.
- Write `.env` with confirm mode still returns `deny`.
- Delete `.env` returns `deny`.
- Rename with sensitive source returns `deny`.
- Rename with sensitive target returns `deny`.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## Recommendation

Keep B1 as a minimal safety patch. Do not wire this policy into UI, Working Files, Composer Attachments, CLI, SDK, or file tools until the next scoped phase defines storage and user-facing confirmation flows.
