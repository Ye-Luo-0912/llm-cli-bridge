# V2.14.0-I - Real File Tool Execution / Claude Read Handoff / Safe Search Report

## FileToolExecutor

`src/fileToolExecutor.ts` adds a real executor for read-only file tools.

- `stat` returns size, mtime, file/directory flags, and fileType.
- `read` reads authorized text-like files with byte and character limits.
- `list` lists Vault or authorized directories with entry and depth limits.
- `search` searches Vault or authorized directories with file, result, byte, and extension limits.
- No write/delete/rename/diff execution is implemented.

## PolicyGateFlow

Every operation calls `evaluateFileToolPolicy` before touching the filesystem.

- `allow` executes the operation.
- `confirm` returns a pending request and does not read/list/search.
- `deny` returns a refusal result and does not touch file content.
- Unauthorized external `read` and `stat` can create pending external read requests.
- Unauthorized external `list` and `search` stay denied and do not create pending requests.

## BoundedRead

Read content remains bounded.

- Text, markdown, json, log, and similar extensions can be read only within max byte/char limits.
- Oversized reads return truncated content and explicit truncation metadata.
- Sensitive paths are denied.
- Image, PDF, binary, and unknown files do not get parsed or injected as content.

## ListSearchLimits

Directory traversal is bounded.

- `list` has max entry and max depth limits.
- `search` has max file count, max result count, per-file byte limit, and extension allowlist.
- Sensitive paths are skipped or denied.
- Search does not scan arbitrary binary/PDF/image data.

## ClaudeReadHandoff

Refs-only files return an explicit handoff hint.

- Image/PDF/binary/unknown read requests return `readMode=refs-only`.
- The result tells the agent it can ask Claude Code to use `Read` on the path if inspection is needed.
- The plugin does not OCR, parse PDFs, decode images, or base64 files into prompts.

## WorkingSetIntegration

Runtime integration remains scoped.

- `LLMBridgeView.executeFileToolRequest()` calls the executor with current policy and Working Set refs.
- Confirm results enqueue pending external read requests in the existing pending UI.
- Approved external files can become active FileRefs through the existing approval flow.
- Pending/denied refs remain excluded from the prompt FileRef index.

## Tests

Added coverage verifies:

- `stat` allowed/confirm/deny.
- `read` allowed bounded content.
- Oversized read truncation.
- Image/PDF/binary refs-only handoff.
- Authorized directory listing.
- Unauthorized external list denial.
- Authorized search with limits.
- Sensitive path denial.
- External pending approval flow.
- Prompt package excludes unauthorized external content.
- Executor contains no write/delete/rename implementation.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

- This phase exposes a real executor and Bridge View caller, but does not yet wire Claude/SDK tool calls directly into it.
- Search is intentionally text-extension based and not a semantic index.
- Large-file behavior is bounded truncation for text-like files and refs-only for non-text files.

## Recommendation

Freeze V2.14.0-I if build/unit/process pass and backend diffs remain absent. The next phase can connect agent tool-call routing to `executeFileToolRequest()` without changing AgentEvent v0.1.
