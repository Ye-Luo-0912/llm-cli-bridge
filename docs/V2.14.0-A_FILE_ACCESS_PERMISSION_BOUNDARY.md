# V2.14.0-A File Access Permission Boundary Report

> Goal: define the file access permission boundary for future Working Files, file refs, file tools, and diff/write flows. This phase is boundary design plus audit only. It does not change runtime behavior, AgentEvent, Skills runtime, CLI/SDK mainline, or UI architecture.

## CurrentState

Current file access is Vault-centric.

- `view.ts` reads the current active Obsidian note through `app.vault.read(activeFile)` only when `includeActiveNote` is enabled, and reads editor selection directly from the active Markdown view.
- `promptPackage.ts` accepts only `activeFileContent` and `selection` as inline content inputs. It has no external file list, no external file reader, and no Agent Skill/file tool injection path.
- `state.ts` writes run state into `.llm-bridge/state/` under the Vault root: `current.json`, optional `active-note.md`, optional `selection.md`, and optional `metadata.json`.
- `fileDiff.ts` scans only the Vault root for Markdown files and excludes `.obsidian`, `.llm-bridge`, `node_modules`, `.git`, `LLM-AgentRuntime`, `dist`, and `build`.
- `actions.ts` accepts only Vault-relative paths. Absolute paths and path traversal are rejected before execution. Write actions are limited to Obsidian Vault APIs: `create_note`, `append_to_note`, `insert_at_cursor`, and `replace_selection`.
- `permissionPolicy.ts` currently classifies action bridge operations as low read or medium write. It notes Vault-outside, `.obsidian`, env, shell, and delete as high-risk future extensions.
- `sdkPermission.ts` classifies absolute paths, `.obsidian`, `.env`, `.git`, `.llm-bridge` credentials, shell, delete, and network as high-risk. SDK `canUseTool` is only active in `sdk-experimental`.
- CLI/auto backend receives a final prompt and Vault cwd. It has no file permission model beyond the process cwd and existing Claude Code permissions.
- SDK backend attaches `canUseTool` separately from Skills options. Tool permissions remain independent from Agent Skills.

Current gaps:

- There is no first-class `readRoots` / `writeRoots` setting yet.
- There is no explicit external read allowlist.
- There is no shared file access policy module for Working Files or future file tools.
- Existing prompt package is inline-content based and should not be extended by blindly concatenating external file bodies.

UI debt:

- Agent Skills UI polish should not continue in V2.14-A. Remaining visual consistency work belongs in a later unified UI Refactor debt item.

## ReadPolicy

Target policy:

- `readRoots` defaults to `[vaultPath]`.
- Users may explicitly add external directories to `readRoots`.
- External paths are read-only by default. They may be used for file refs, metadata, search, and explicit on-demand reads.
- External file reads must be explicit and bounded:
  - Prefer file refs and working-set entries over inline content.
  - Read content only when the agent/user requests that file.
  - Apply size limits, truncation, and sensitive-path checks before including content in any prompt context.
- Reads outside `readRoots` are denied.
- Reads inside sensitive paths are denied by default or require strong confirmation depending on path class.

Path resolution rules:

- Normalize and resolve all candidate paths before policy decisions.
- Resolve symlinks where feasible before deciding root membership.
- Treat Windows drive-letter paths and POSIX absolute paths consistently.
- Root membership must use path containment, not string prefix alone.
- Vault-relative paths are resolved against `vaultPath`.
- External file refs must retain both display path and resolved absolute path internally.

Prompt boundary:

- Do not append external file content wholesale to `promptPackage`.
- Future prompt construction should include compact file refs, summaries, or selected snippets.
- Full external file content should be read lazily and bounded by explicit max size / max chars.

## WritePolicy

Target policy:

- `writeRoots` defaults to:
  - `vaultPath`
  - resolved `outputDir` when it is inside `vaultPath`
- Vault outside paths are not writable by default.
- External paths may be readable through `readRoots`, but they are not writable, deletable, or renameable unless a future explicit write root model is designed and approved.
- Vault-outside write, delete, and rename operations are denied, not merely high-risk.
- Vault-internal writes remain controlled by action confirmation and `permissionPolicy`.
- `.llm-bridge/` internal writes are plugin-owned only. Agent/tool initiated writes to bridge credentials/config/log internals remain blocked.
- `.claude/skills/` writes remain governed by Agent Skills materialization ownership rules and are not generalized file-write permission.

Allowed write classes for current runtime:

- Obsidian note writes through action bridge:
  - `create_note`
  - `append_to_note`
  - `insert_at_cursor`
  - `replace_selection`
- Plugin-owned state/log/session/skill metadata writes under `.llm-bridge/`.
- Agent Skill materialization through `agentSkills.ts` with plugin marker/hash checks.

Denied write classes:

- Absolute external filesystem paths.
- Path traversal outside the Vault.
- Delete/rename outside the Vault.
- Any write to user credential/config paths.
- Any general-purpose file write outside `writeRoots`.

## SensitivePathPolicy

Default deny or strong confirmation paths:

- `.env`, `.env.*`
- any path segment or filename containing credential material such as `token`, `credentials`, `secrets`, `password`, `api_key`, `apikey`
- `.ssh/`
- private keys or key material:
  - `id_rsa`, `id_ed25519`, `*.pem`, `*.key`, `*.p12`, `*.pfx`
  - content marker `BEGIN ... PRIVATE KEY`
- `.git/config`
- `.git/` internals generally
- `.obsidian` internal configuration:
  - workspace files
  - plugin manifests/settings
  - app configuration
- `.llm-bridge/bridge.json`, token, config, credential, and secret files

Policy by operation:

- Read:
  - Deny by default for private keys and credential stores.
  - Strong confirmation for `.obsidian` internal config if a future read feature needs it.
  - Deny `.git/config` by default because it can contain remotes, usernames, and tokens.
- Write:
  - Deny outside `writeRoots`.
  - Deny all sensitive paths even inside the Vault unless the plugin itself owns that file and the operation is a known plugin-maintenance write.
- Delete/rename:
  - Deny outside the Vault.
  - Deny sensitive paths inside the Vault by default.

Existing source alignment:

- `actions.ts` already rejects absolute paths, traversal, `.obsidian`, `.env`, `.git`, `token`, `secrets`, and `credentials` for action paths.
- `sdkPermission.ts` already flags absolute paths, `.obsidian`, `.env`, `.git`, bridge credentials, shell, delete, and network as high-risk.
- `fileDiff.ts` already excludes `.obsidian`, `.llm-bridge`, and `.git` from generated-file detection.

## BackendImpact

No runtime changes in V2.14-A.

CLI/auto:

- No CLI args change.
- No change to cwd.
- No new file events.
- Future Working Files should pass refs through prompt/SDK options only after policy checks; it should not change the CLI mainline contract in this phase.

SDK:

- No change to `buildSdkOptions()`, `settingSources`, `skills`, or `canUseTool` in this phase.
- Future SDK file tools must use the shared file access policy before allowing SDK tool execution or exposing file content.
- `canUseTool` remains the permission gate for tool use; file access policy becomes an input to risk assessment and allow/deny decisions.

AgentEvent:

- `AgentEvent v0.1` remains unchanged.
- Do not add file/tool events in this phase.
- UI-only workflow events remain separate from AgentEvent.

Skills runtime:

- Do not change Agent Skills manifest/materialization.
- Do not treat Skills as file-access grants.
- Skills may describe capabilities, but file access is still governed by read/write roots and permission policy.

## ImplementationPhases

### Phase A: Boundary Report and Tests

- Add this report.
- Add source-level tests that assert the boundary:
  - read roots default to Vault plus explicit external additions.
  - write roots default to Vault and Vault-contained outputDir.
  - external paths are read-only.
  - sensitive paths are denied or require strong confirmation.
  - prompt package has no external full-file ingestion.
  - AgentEvent, Skills runtime, CLI/SDK mainline remain unchanged.

### Phase B: Shared Policy Module

- Add `src/fileAccessPolicy.ts`.
- Define:
  - `FileAccessSettings`
  - `ResolvedRoot`
  - `FileAccessOperation = read | write | delete | rename`
  - `FileAccessDecision = allow | confirm | deny`
- Implement path normalization, containment, root resolution, and sensitive-path checks.
- Add unit tests for Windows/POSIX paths, symlink-aware containment where feasible, and outputDir normalization.

### Phase C: Settings Storage

- Add settings fields:
  - `readRoots: string[]`
  - `writeRoots: string[]`
- Default `readRoots` to `[vaultPath]` at runtime.
- Default `writeRoots` to `[vaultPath, outputDirInsideVault]`.
- UI should allow explicit external read roots only.
- Defer external write roots unless separately approved.

### Phase D: Working Files / File Refs

- Add a working-set model that stores refs, not full content.
- Add explicit read action for selected refs with truncation and sensitive checks.
- Surface file refs in prompt as paths and metadata first.
- Only include content when explicitly selected and bounded.

### Phase E: File Tools / Diff Writes

- Route all file tool operations through `fileAccessPolicy`.
- Continue denying external writes/deletes/renames by default.
- Keep Vault writes confirmable through existing permission policy.
- Keep generated-file diff scoped to Vault unless a future external workspace diff model is explicitly designed.

## Risks

- Path containment is easy to get wrong on Windows if comparisons use raw string prefixes.
- Symlinks can escape an allowed root unless resolved carefully.
- External reads can leak secrets if sensitive paths are not checked before content inclusion.
- If outputDir is allowed as a raw string, `../` or absolute paths could accidentally expand write scope.
- SDK tool calls may use different path parameter names; policy integration must inspect all relevant path fields.
- Prompt bloat risk increases if external file bodies are concatenated into promptPackage.
- User expectations may be confused if external files are visible but not writable; UI copy must be explicit.

## Recommendation

Proceed to V2.14.0-B with a shared `fileAccessPolicy.ts` module before implementing Working Files or file tools.

Acceptance boundary for the next phase:

- External read roots are explicit and read-only.
- Write roots are Vault-scoped by default.
- Sensitive paths are denied or strongly confirmed.
- External file content is not blindly appended to promptPackage.
- Backend mainline and AgentEvent stay unchanged.
