# V2.16-F - Assistant Turn Process + Markdown Output

## Goal

Move runtime process/thinking from a separate global debug area into the assistant turn itself. The user should see continuous progress while the agent is running, and the process should remain available after completion without competing with the final answer.

## Assistant Turn Layout

- Header: runtime label, status, timestamp.
- Process: shown above the final answer in the same assistant turn.
- Final answer: rendered through Obsidian `MarkdownRenderer`.

Running turns keep the process area visible. Completed turns collapse the process area by default, but keep it expandable. Failed turns can still expose error detail without showing raw logs in normal user mode.

## Process / Thinking Policy

- Display SDK/CLI-provided thinking summaries and workflow/tool events.
- Do not invent hidden chain-of-thought.
- If no runtime event has arrived yet, show an active "starting/waiting for first event" placeholder so the UI does not look frozen.
- Developer mode keeps raw command/workflow/log details available.

## Global Run Flow

The legacy global Run Flow panel is no longer rendered in default user mode. It remains available in Developer mode for diagnostics. User-facing process now belongs to the message that produced it.

## Markdown Output

Assistant content is rendered with Obsidian Markdown rendering instead of plain `textContent`, so headings, lists, code blocks, and links display like normal Obsidian content.

## Tests

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## Remaining Risk

SDKs may provide sparse thinking events for some models or modes. The UI now preserves and surfaces whatever public process events are available, while the initial placeholder prevents a silent wait before the first event.
