# V2.13.0 Skills Runtime Alignment Plan

> Goal: align Skills with the Claude runtime model. A Skill is an agent-discoverable and agent-callable capability, not a composer prompt injector. This phase is design plus minimum implementation plan only.

## 0. Constraints

- Do not change `AgentEvent v0.1`.
- Do not add tool events to `AgentEvent`.
- Do not change the CLI/auto mainline contract.
- Keep `sdk-experimental` disabled by default.
- Do not remove legacy prompt insertion abruptly. Rename and isolate it as Prompt Snippet behavior.

## 1. Audit

### Storage

Current files:

- `.llm-bridge/skills.md`: legacy Markdown bundle. It stores `## title`, one description line, then prompt body.
- `.llm-bridge/skills/*.md`: legacy imported single-skill files using the same format.
- `.llm-bridge/skills-state.json`: per-skill UI metadata keyed by skill name: `pinned`, `applyCount`, `lastUsedAt`, `groupOverride`, `lastCombo`.

Current problem:

- The storage format calls these records Skills, but the body is treated as prompt text for the composer.
- There is no runtime materialization into `.claude/skills/<slug>/SKILL.md`.
- There is no first-class distinction between agent capabilities and reusable prompt snippets.

### `skills.ts`

Current behavior:

- `Skill` has `name`, `description`, `prompt`, `tags`.
- `parseSkillsMarkdown()` parses prompt body into `Skill.prompt`.
- `expandSkillPrompt()`, `scanSkillPrompt()`, and `truncateSkillPrompt()` exist specifically for prompt injection.
- `importSkillFromText()` and `updateImportedSkill()` write legacy `.llm-bridge/skills/*.md` files.

Current problem:

- `Skill.prompt` is the wrong center of gravity for Agent Skills.
- The module mixes three concerns: parsing legacy prompt snippets, metadata extraction, and runtime Skill authoring.

### `skillsState.ts`

Current behavior:

- Tracks UI metadata and usage counters.
- Usage means "inserted/applied into composer", not "available to agent" or "invoked by agent".

Current problem:

- `applyCount` and `lastUsedAt` become ambiguous once Agent Skills are runtime capabilities. They should mean explicit UI selection/enablement, not actual model invocation, unless the runtime exposes invocation telemetry separately.

### `view.ts`

Current behavior:

- Skills panel renders search, group, sort, enable/disable, pin, preview, import, edit, delete.
- The main actions are prompt insertion:
  - `insertSkillAtCursor()`
  - `appendSkillToInput()`
  - `applyCombo()`
  - older `applySkill()`
- Status bar counts `appliedSkillNames`, meaning prompt snippets inserted this run.
- `run()` builds prompt via `buildPromptPackage(userInput, snapshot, settings)` and does not pass selected runtime Skills.

Current problem:

- The UI owns deep prompt composition for Skills.
- Agent Skills are invisible to the actual Claude runtime unless their text is pasted into the user prompt.
- Labels such as Apply Skill and Combo Apply encode the wrong mental model.

### CLI backend

Current behavior:

- `ClaudeCliBackend` receives a final `task.prompt` and runs `claude` via `spawn`.
- `commandProfile.ts` builds base args plus `--continue`, `--resume`, `--permission-mode`, and extra args.
- Claude Code filesystem settings are not managed by the plugin.

Current implication:

- CLI/auto can rely on Claude Code discovering `.claude/skills/*/SKILL.md` from the working directory.
- Optional direct invocation can be represented in the user prompt as `/skill-name ...`, but the backend should not paste Skill bodies into the prompt.

### SDK backend

Current behavior:

- `buildSdkOptions()` sets `cwd`, `permissionMode`, `includePartialMessages`, session flags, and `extraArgs`.
- `runRealSdkQuery()` attaches `canUseTool`.
- No `settingSources` or `skills` option is currently set.
- Tool permissions are controlled by `allowedTools`/`canUseTool` style logic, not by Skill files.

Current implication:

- SDK support needs two independent controls:
  - discovery through `settingSources` that includes project filesystem settings
  - filtering through `skills`
- Tool permissions must remain controlled by the existing permission model.

## 2. Correct Responsibility Boundary

### Agent Skill

An Agent Skill is a runtime capability discoverable by Claude. It is stored as:

```text
.claude/
  skills/
    <slug>/
      SKILL.md
```

It contains:

```markdown
---
name: <stable-name>
description: <when Claude should use it>
---

# <Human title>

<instructions>
```

Rules:

- Agent Skill content is not inserted into the composer.
- The model decides when to load/use it, or the user explicitly asks for it with `/skill-name`.
- UI can enable/disable a Skill for the current run.
- UI can edit, preview, pin, search, and materialize Skills.
- UI must not concatenate Skill instructions into `task.prompt`.

### Prompt Snippet

A Prompt Snippet is reusable text for the composer.

Rules:

- It may be inserted into the composer explicitly.
- It should be named Prompt Snippet in UI and storage.
- UI commands should say `Insert prompt`, not `Apply Skill`.
- Prompt Snippets may keep `{{outputDir}}` expansion, sensitive scan, and truncation.
- Prompt Snippets must not be passed to SDK as `skills`.

## 3. Skill Materialization Path

Source of truth in V2.13 minimum implementation:

- Legacy `.llm-bridge/skills.md` and `.llm-bridge/skills/*.md` remain readable.
- Each legacy item is classified during migration:
  - Agent Skill candidate: has capability-style description/instructions and should be discoverable by Claude.
  - Prompt Snippet: explicit prompt template intended for composer insertion.

Materialized output:

```text
<vault>/.claude/skills/<slug>/SKILL.md
```

Slug rules:

- Lowercase ASCII where possible.
- Replace whitespace and illegal path characters with `-`.
- For CJK names, use a stable sanitized fallback plus short hash, e.g. `skill-<hash>`, unless user edits slug.
- Slug must not change when display name changes unless explicitly renamed.

`SKILL.md` mapping:

- `name`: stable runtime name, preferably slug or explicit runtime name.
- `description`: concise trigger description used by Claude to decide when to invoke it.
- Markdown body: instructions, migrated from legacy prompt body after removing composer-specific phrasing where possible.

Important:

- `.claude/skills/` is the runtime-facing materialized directory.
- `.llm-bridge/` remains plugin-owned storage and migration state.

## 4. StorageMigration

Add a migration module, proposed `src/skillsMigration.ts`.

New/changed storage:

- `.llm-bridge/skills.md`: legacy Prompt Snippet bundle; do not overwrite.
- `.llm-bridge/skills/*.md`: legacy imported Prompt Snippet or Skill source; do not delete during V2.13.
- `.llm-bridge/agent-skills.json`: plugin-managed manifest for Agent Skills.
- `.claude/skills/<slug>/SKILL.md`: runtime materialization target.

Proposed manifest:

```ts
interface AgentSkillRecord {
  id: string;
  slug: string;
  name: string;
  description: string;
  instructions: string;
  enabled: boolean;
  source: "migrated" | "imported" | "manual";
  sourcePath?: string;
  materializedPath: string;
  updatedAt: string;
}
```

Migration steps:

1. Read legacy skills with current parser.
2. Create Prompt Snippet records by default to preserve behavior.
3. Offer "Convert to Agent Skill" for selected records.
4. Materialize converted records into `.claude/skills/<slug>/SKILL.md`.
5. Persist selected runtime Skill names/slugs separately from prompt snippets.

Minimum V2.13 behavior:

- No automatic destructive migration.
- No deletion of `.llm-bridge/skills.md`.
- Agent Skill conversion is explicit.
- Existing legacy data still renders, but under Prompt Snippets.

## 5. UIChanges

Rename and split UI surfaces:

- `Skills` tab/panel becomes `Agent Skills`.
- Legacy prompt injection surface becomes `Prompt Snippets`.
- Existing `Apply Skill`, `Append`, and `Combo Apply` become:
  - `Insert prompt`
  - `Insert at cursor`
  - `Insert selected snippets`

Agent Skills UI responsibilities:

- List runtime-capable Skills.
- Enable/disable per Skill.
- Select available Skills for this run.
- Preview generated `SKILL.md`.
- Edit name, description, instructions, slug.
- Materialize/re-materialize to `.claude/skills/<slug>/SKILL.md`.
- Show materialization status: missing, stale, current.

Agent Skills UI non-responsibilities:

- No deep prompt concatenation.
- No direct insertion of Agent Skill instructions into composer.
- No hidden replacement of user input.

Status bar changes:

- Replace "Applied Skills" with "Skills available" or "Runtime Skills".
- Prompt snippet usage can be tracked separately as "Snippets inserted".

## 6. BackendCapabilities

### CLI/auto

Expected behavior:

- Keep the existing CLI/auto mainline.
- Before run, ensure enabled Agent Skills are materialized under `<cwd>/.claude/skills/`.
- Let Claude Code discover Skills through its normal filesystem settings.
- Do not modify `AgentEvent`.
- Do not add tool events.

Optional direct invocation:

- If user selects exactly one Agent Skill and chooses direct invocation, prefix the user request with `/skill-name`.
- This is a command-style invocation hint, not instruction-body injection.
- Keep this as an explicit user action, not default behavior.

### SDK

Expected behavior:

- `buildSdkOptions()` should include:
  - `settingSources: ["user", "project"]` or the SDK-supported equivalent
  - `skills: "all"` or a list of enabled runtime Skill names
- Use `skills: []` when no Agent Skills should be available.
- Keep tool permissions in `allowedTools`/`canUseTool`; do not rely on Skill frontmatter for SDK permissions.
- Do not paste Skill instructions into `prompt`.

Reference facts from Claude docs:

- SDK Skills are filesystem artifacts under `.claude/skills/*/SKILL.md`.
- They are loaded through filesystem setting sources.
- The SDK `skills` option controls which discovered Skills are available.
- SDK tool access remains controlled through `allowedTools`/`canUseTool`; Skill frontmatter `allowed-tools` is CLI-only.

## 7. Compatibility

Backward compatible behavior:

- Existing `.llm-bridge/skills.md` remains readable.
- Existing `.llm-bridge/skills/*.md` remains editable.
- Existing prompt insertion survives under Prompt Snippets.
- Existing `skills-state.json` can continue to track UI metadata, but meaning must be scoped:
  - Prompt Snippet usage for legacy snippets
  - Agent Skill selection/materialization metadata for runtime Skills

Breaking changes to avoid:

- Do not suddenly stop rendering existing user data.
- Do not delete legacy files.
- Do not change default backend mode.
- Do not inject migrated Skill instructions into every prompt.

Compatibility bridge:

- Legacy `Skill` type can be renamed internally later.
- Minimum V2.13 may introduce new types while leaving old parser intact:

```ts
type SkillKind = "agent-skill" | "prompt-snippet";
```

## 8. Tests

Storage tests:

- Parse legacy `.llm-bridge/skills.md` into Prompt Snippets.
- Convert one legacy item into Agent Skill.
- Generate deterministic slug and `SKILL.md`.
- Re-materialization updates stale files.
- No migration deletes legacy files.

UI tests:

- Agent Skill selection does not change composer text.
- Prompt Snippet `Insert prompt` changes composer text.
- Legacy labels no longer say `Apply Skill`.
- Agent Skill enabled/disabled state persists.
- Preview shows generated `SKILL.md` content.

CLI tests:

- Auto/CLI command args remain unchanged by default.
- Materialized `.claude/skills/<slug>/SKILL.md` exists before run.
- Direct invocation mode adds `/skill-name` only when explicitly selected.
- `AgentEvent v0.1` remains unchanged.

SDK tests:

- `buildSdkOptions()` includes `settingSources` and `skills`.
- `skills` list follows enabled Agent Skills.
- `skills: []` when disabled.
- `canUseTool` remains attached and permission decisions still flow through existing logic.
- `sdk-experimental` remains default off.

Regression tests:

- Existing prompt package still includes active note/selection/output rules/user request.
- Prompt Snippet sensitive scan and truncation remain intact.
- Existing session serialization does not leak Skill instructions unexpectedly.

## 9. Risks

- Skill discovery depends on correct `cwd`; if the Vault root is not the runtime cwd, `.claude/skills/` may not be found.
- SDK option names can drift; gate SDK Skills support with diagnostics and tests.
- Legacy Skills may contain composer-specific instructions that make poor Agent Skills without human cleanup.
- Converting every legacy item automatically would create noisy Skills and degrade model choice.
- The word "Skill" is already used throughout UI/tests; partial rename risks confusing users unless labels are updated consistently.
- `.claude/skills/` may be user-managed; plugin writes should be explicit and should avoid overwriting non-plugin files.

## 10. ImplementationPhases

### Phase 1: Types and Storage Boundary

- Add `AgentSkillRecord` and `PromptSnippet` types.
- Add slug generation and `SKILL.md` serializer.
- Add `agent-skills.json` load/save helpers.
- Keep existing legacy parser untouched.

### Phase 2: Explicit Conversion and Materialization

- Add conversion from legacy item to Agent Skill.
- Write `.claude/skills/<slug>/SKILL.md`.
- Add stale/current detection.
- Add tests for materialization.

### Phase 3: UI Rename With Minimal Layout Change

- Rename legacy panel labels:
  - `Skill` prompt body -> `Prompt Snippet`
  - `Apply Skill` -> `Insert prompt`
  - `Combo Apply` -> `Insert selected snippets`
- Add an Agent Skills list for enable/disable/preview/materialize.
- Do not redesign layout yet.

### Phase 4: CLI Runtime Integration

- Ensure selected/enabled Agent Skills are materialized before run.
- Keep CLI args unchanged.
- Add optional explicit `/skill-name` invocation path.

### Phase 5: SDK Runtime Integration

- Extend `buildSdkOptions()` with `settingSources` and `skills`.
- Keep `canUseTool` and permission behavior unchanged.
- Add diagnostics when SDK cannot find materialized Skills.

### Phase 6: Cleanup and Follow-up UI

- Remove misleading "applied skill" status language.
- Split usage counters into snippet insertion and Skill availability.
- Only after runtime correctness is locked, optimize UI.

## 11. Minimum V2.13 Acceptance Criteria

- Agent Skill instructions are no longer directly appended to composer or deep-concatenated into the prompt package.
- Prompt insertion remains available only under Prompt Snippet / Insert prompt labels.
- At least one Agent Skill can be materialized to `.claude/skills/<slug>/SKILL.md`.
- CLI/auto continues to run with the same base behavior.
- SDK options have a planned and tested path for `settingSources` plus `skills`.
- Tool permissions remain controlled by existing permission mechanisms.
