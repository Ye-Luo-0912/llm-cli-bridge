# V2.14.0-L - CLI/SDK Native Handoff Simplification Report

## NativeHandoffStrategy

The plugin now treats Claude Code / Claude SDK as the native file execution runtime.

- Vault-internal ordinary file reads and edits should be handled by Claude Code or SDK native file capabilities.
- The plugin provides Vault context, Working Set metadata, attachment paths, and permission boundary guidance.
- Image, PDF, binary, and unknown attachments remain refs. The handoff tells the agent to use Claude Code Read or SDK-native capability instead of plugin-side parsing.

## PluginResponsibilityBoundary

The plugin remains a context and permission bridge, not a custom filesystem runtime.

- Keep FileRef, Working Set, bounded user attachment ingestion, external read grants, and read-only file policy baseline.
- Do not add a Vault write executor in this phase.
- Do not add backup, rollback, audit transaction, SDK image streaming, PDF parsing, OCR, or base64 image injection.
- Do not expand runtime tool registration beyond the existing read-only policy gate.

## VaultOperationGuidance

`promptPackage` now includes explicit native handoff guidance:

- The current Vault root is the working directory for the task.
- Claude Code / SDK may reasonably read, create, or edit ordinary files inside the Vault.
- External paths must not be written, deleted, or renamed.
- Sensitive paths must not be modified.
- FileRef and attachment paths are provided as metadata and bounded snippets only where already allowed.

## ExternalBoundary

Existing external path boundaries remain unchanged.

- External read still requires user authorization through the existing pending/session grant model.
- External write, delete, and rename remain hard denied by `fileAccessPolicy`.
- Sensitive paths continue to deny writes/deletes/renames and use deny or strong-confirm behavior for reads.

## SimplificationDecision

V2.14-L intentionally stops the self-hosted file runtime expansion.

The previous read-only executor, route, and runtime adapter remain as stable policy-gated baseline code, but future file execution should prefer native Claude Code / SDK handoff unless a separate, explicitly scoped need is approved.

## Tests

Covered by `scripts/run-tests.mjs`:

- Prompt/handoff contains CLI/SDK native file operation guidance.
- External write/delete/rename remain denied.
- Sensitive path write remains denied while sensitive read can still be confirm-gated.
- Read-only baseline and runtime adapter remain available.
- Runtime adapter still has no write/delete/rename route and no direct write implementation.

Validation commands:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## Recommendation

Freeze V2.14-L as the responsibility boundary reset.

Next work should focus on UX/handoff clarity and native runtime integration points, not additional plugin-owned filesystem execution layers.
