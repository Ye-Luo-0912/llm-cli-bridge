# V2.14.0-E - Pending External Read Approval Runtime UI Report

## RuntimeStore

V2.14.0-E wires the V2.14-D grant model into `LLMBridgeView` runtime state.

- `externalReadGrantStore` is held on the Bridge View instance.
- The store is session-scoped and in-memory only.
- New session, restored session, and view close clear pending requests and session grants.
- No permanent external read root setting is added.

## ApprovalUI

The Chat panel now includes an `External Read Requests` approval panel.

Each pending read request displays:

- `requestedPath`
- `proposedGrantRoot`
- `risk`
- `reason`
- `source`

The UI only manages authorization state. It does not read files, scan directories, or inject content into `promptPackage`.

## GrantActions

The panel exposes three actions:

- `允许本次会话读取此目录`: approves a directory-scope session read grant.
- `仅允许此文件`: approves a file-scope session read grant.
- `拒绝`: removes the pending request without creating a grant.

`queueExternalFileAccessRequest()` only creates pending requests for read operations. Write, delete, and rename requests do not enter the pending queue.

## SafetyBehavior

Safety handling:

- `grantRootSafety=deny`: normal allow buttons are not shown; only reject remains.
- `grantRootSafety=confirm`: buttons use strong-confirm wording and require an explicit user click.
- Sensitive policy remains enforced by `fileAccessPolicy.ts`; directory grants do not bypass sensitive read/write rules.
- External write/delete/rename remain hard denied and are not added to pending requests.

## Tests

The V2.14 policy/UI tests cover:

- Runtime store exists in `LLMBridgeView`.
- Pending UI text and required fields exist.
- Allow directory action calls `approvePendingExternalReadRequest(..., forceFileScope=false)`.
- Allow file action calls `approvePendingExternalReadRequest(..., forceFileScope=true)`.
- Deny action removes pending without creating grants.
- `grantRootSafety=deny` hides normal allow buttons.
- `grantRootSafety=confirm` displays strong-confirm wording.
- Non-read operations cannot create pending requests.
- `promptPackage`, AgentEvent v0.1, Skills runtime, CLI backend, and SDK backend remain unwired.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

- The UI is not connected to a real file tool yet.
- No attachment picker/drop handling is implemented in this phase.
- Pending request creation currently requires future runtime callers to invoke `queueExternalFileAccessRequest()`.
- Project-root inference still depends on safe known marker inputs from a future caller; the policy layer does not scan external directories.

## Recommendation

Next phase can add a narrow runtime caller for external read attempts, still returning only authorization decisions and file references. Do not add file content reads or prompt injection until Working Files/File Refs are explicitly designed.
