# V2.13.0-A Skills Runtime Boundary Audit

> Scope: boundary audit only. This report accepts and narrows the V2.13 Skills Runtime Alignment direction without changing runtime code.

## 0. Baseline

- Branch baseline: current `master`.
- Release tags: do not move `v2.12.0`.
- Tagging: do not create `v2.12.1` or `v2.13.0` in this phase.
- Runtime changes: none in phase A.
- Contract constraints:
  - Do not change `AgentEvent v0.1`.
  - Do not add tool events.
  - Do not change the CLI/auto mainline.
  - Keep `sdk-experimental` disabled by default.

## 1. CurrentMisuse

The current implementation still models `Skill` as a prompt injector.

### Storage

- `.llm-bridge/skills.md` stores a legacy bundle of entries shaped as `## name`, description, and prompt body.
- `.llm-bridge/skills/*.md` stores imported entries using the same legacy format.
- `.llm-bridge/skills-state.json` tracks UI metadata keyed by skill name, including `pinned`, `applyCount`, `lastUsedAt`, `groupOverride`, and `lastCombo`.

These files are plugin-owned legacy storage. They are not Claude runtime Skill files.

### `skills.ts`

Observed behavior:

- `Skill` has `name`, `description`, `prompt`, and `tags`.
- `parseSkillsMarkdown()` parses body text into `Skill.prompt`.
- `expandSkillPrompt()` exists to replace `{{outputDir}}` before insertion.
- `scanSkillPrompt()` and `truncateSkillPrompt()` protect prompt insertion into the composer.
- `importSkillFromText()` and `updateImportedSkill()` write legacy `.llm-bridge/skills/*.md`.

Conclusion:

- Current `Skill` is a reusable prompt template, not an agent-discoverable runtime capability.

### `skillsState.ts`

Observed behavior:

- `recordSkillApplied()` increments `applyCount` and updates `lastUsedAt`.
- `recordCombo()` stores the last prompt-combination order.
- Metadata describes UI use, not Claude runtime discovery or actual model invocation.

Conclusion:

- Current counters are Prompt Snippet usage counters. They must not be reused as "agent invoked this Skill" telemetry.

### `view.ts`

Observed behavior:

- `insertSkillAtCursor()` inserts `Skill.prompt` into the composer.
- `appendSkillToInput()` appends `Skill.prompt` into the composer.
- `applyCombo()` concatenates multiple `Skill.prompt` values into the composer.
- `applySkill()` sets the composer to the expanded prompt.
- Status bar language tracks `appliedSkillNames`, meaning inserted prompt snippets.
- `run()` builds the final prompt from `buildPromptPackage(userInput, snapshot, settings)` and sends it to the backend.

Conclusion:

- The UI currently performs deep Skill prompt composition. This is the behavior V2.13 must stop calling Skill behavior.

### CLI backend

Observed behavior:

- `ClaudeCliBackend` receives `task.prompt` and spawns the configured CLI command.
- `commandProfile.ts` builds command args for `--continue`, `--resume`, `--permission-mode`, and extra args.
- The backend does not manage `.claude/skills`.

Conclusion:

- CLI/auto should continue to rely on Claude Code filesystem discovery. It should not receive Skill instruction bodies through composer concatenation.

### SDK backend

Observed behavior:

- `buildSdkOptions()` sets `cwd`, `permissionMode`, `includePartialMessages`, session args, and extra args.
- `runRealSdkQuery()` attaches `canUseTool`.
- No `settingSources` or `skills` option is currently used.

Conclusion:

- SDK alignment is future work and must be capability gated. It should expose available Agent Skills through SDK options, while tool permissions remain governed by `canUseTool` and related permission code.

## 2. Terminology

### Agent Skill

An Agent Skill is a runtime capability.

Definition:

- Stored as `.claude/skills/<slug>/SKILL.md`.
- Contains `name`, `description`, and instructions.
- Discoverable and callable by Claude runtime.
- May be used automatically by Claude or explicitly invoked with `/skill-name`.
- Must not be inserted into the composer.
- Must not be deep-concatenated into `task.prompt`.

### Prompt Snippet

A Prompt Snippet is a legacy prompt template.

Definition:

- Stored in legacy `.llm-bridge/skills.md` or `.llm-bridge/skills/*.md` until migrated.
- Can be explicitly inserted into the composer.
- UI command should be named `Insert prompt`.
- Can retain `{{outputDir}}` expansion, sensitive scanning, and truncation.
- Must not be passed to SDK as a `skills` option.

## 3. StorageBoundary

### Plugin-owned legacy storage

These paths remain plugin-owned and backward compatible:

- `.llm-bridge/skills.md`
- `.llm-bridge/skills/*.md`
- `.llm-bridge/skills-state.json`

V2.13-A interpretation:

- All legacy entries are Prompt Snippets by default.
- Existing user content must remain readable.
- No automatic conversion to Agent Skills.
- No destructive rewrite.

### Plugin-owned Agent Skill manifest

Planned path:

- `.llm-bridge/agent-skills.json`

The manifest must track all plugin-managed runtime Skills:

```ts
interface AgentSkillRecord {
  id: string;
  slug: string;
  name: string;
  description: string;
  instructions: string;
  enabled: boolean;
  source: "manual" | "converted" | "imported";
  sourcePath?: string;
  materializedPath: string;
  materializedHash: string;
  updatedAt: string;
}
```

### Runtime materialization

Runtime path:

- `.claude/skills/<slug>/SKILL.md`

Write rule:

- Only write files tracked by `.llm-bridge/agent-skills.json`.
- Do not overwrite an existing untracked `.claude/skills/<slug>/SKILL.md`.
- Add a plugin marker in generated files.
- Compare marker/hash before updating.
- If a tracked generated file was manually edited, mark it stale/conflicted instead of overwriting silently.

Proposed generated file marker:

```markdown
<!-- generated-by: llm-cli-bridge -->
<!-- source-id: <agent-skill-id> -->
<!-- source-hash: <hash> -->
```

## 4. MigrationPolicy

Default migration policy:

- Do not auto-migrate legacy prompt templates into Agent Skills.
- Treat every existing `.llm-bridge/skills.md` and `.llm-bridge/skills/*.md` entry as Prompt Snippet first.
- Let users explicitly convert a Prompt Snippet into an Agent Skill.
- Preserve legacy files after conversion.
- Materialize Agent Skills only after explicit user action or explicit manifest creation.

Why:

- Many legacy entries contain imperative prompt text such as "please summarize current note" or "write output to {{outputDir}}".
- Those are not necessarily good runtime Skill descriptions.
- Automatic migration would create noisy runtime capabilities and degrade Claude's ability to choose the right Skill.

## 5. BackendCapabilityRisks

### CLI/auto risks

- Skill discovery depends on `cwd` being the Vault root.
- `.claude/skills` discovery is owned by Claude Code, not the plugin.
- Direct `/skill-name` invocation should be explicit and should not paste instructions.
- Changing CLI args could regress the stable auto path, so V2.13 must keep CLI args unchanged by default.

### SDK risks

- SDK support depends on the installed SDK version supporting `settingSources` and `skills`.
- Option names and semantics can drift across SDK versions.
- `skills` availability is independent from tool permission.
- Skill frontmatter such as `allowed-tools` must not bypass `canUseTool`.

SDK alignment rule:

- Gate SDK Skill options behind capability detection.
- If unsupported, omit `skills`/`settingSources` and report diagnostics.
- Keep `sdk-experimental` default off.

### UI risks

- Existing labels teach users that "Skill" means prompt insertion.
- A partial rename could leave the wrong model in place.
- Status counters such as "applied skills" will become misleading unless split into Prompt Snippet insertion and Agent Skill availability.

## 6. ImplementationPhases

### A: Boundary audit

Status: this report.

Outputs:

- Confirm current misuse.
- Freeze terminology.
- Define storage and runtime boundaries.
- Define non-destructive migration policy.

No runtime code changes.

### B: Legacy UI rename to Prompt Snippets

Goal:

- Stop calling composer prompt templates Skills.

Work:

- Rename UI labels from Skill prompt insertion to Prompt Snippet.
- Rename `Apply Skill` behavior to `Insert prompt`.
- Rename combo action to `Insert selected snippets`.
- Keep legacy storage readable.
- Keep existing prompt insertion behavior under the new name.

Acceptance:

- No user-facing legacy prompt insertion path says "Apply Skill".
- Agent Skill terminology is reserved for runtime materialized capabilities.

### C: Agent skill manifest + materialization

Goal:

- Add plugin-managed Agent Skill storage and `.claude/skills` materialization.

Work:

- Add `AgentSkillRecord`.
- Add slug generation.
- Add `SKILL.md` serializer.
- Add `.llm-bridge/agent-skills.json`.
- Add write rules with marker/hash conflict protection.
- Add explicit Convert to Agent Skill action.

Acceptance:

- A plugin-managed Agent Skill materializes to `.claude/skills/<slug>/SKILL.md`.
- Untracked `.claude/skills` files are not overwritten.

### D: CLI runtime alignment

Goal:

- Let Claude Code discover materialized Agent Skills.

Work:

- Before run, ensure enabled Agent Skills are materialized.
- Keep CLI command and args unchanged by default.
- Add optional explicit `/skill-name` invocation mode only as a user-selected path.

Acceptance:

- CLI/auto mainline remains stable.
- No Agent Skill instruction body is appended to the composer or prompt package.

### E: SDK capability-gated alignment

Goal:

- Expose materialized Agent Skills to SDK only when supported.

Work:

- Detect SDK capability for `settingSources` and `skills`.
- When supported, pass project filesystem settings and enabled Skill names.
- Keep `canUseTool` permission logic unchanged.
- Add diagnostics when SDK Skill control is unavailable.

Acceptance:

- `sdk-experimental` remains default off.
- Tool permissions still flow through existing permission controls.
- No new `AgentEvent` or tool event is added.

## 7. Recommendation

Proceed with V2.13 in the defined phases.

Immediate next step should be phase B:

- Rename the legacy UI and actions to Prompt Snippets.
- Keep behavior intact.
- Do not add runtime Skill materialization in the same step.

Reason:

- The current user-facing terminology is actively wrong.
- Renaming first reduces confusion before adding the real Agent Skill runtime path.
- It preserves the CLI/auto mainline while creating room for `.claude/skills` materialization in phase C.
