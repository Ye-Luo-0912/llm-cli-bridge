## 1. RootCause（根因分析）

**Blocker**: 导入 Skill 重命名后 `pinned/applyCount/lastUsedAt/groupOverride` 未迁移到新名称。

**根因（两层，CDP 验证后确认）**:

**第一层（V2.12.1 第一次修复发现）**: `view.ts` 中 `openEditSkillDialog` 的时序冲突：
1. `renameSkillMeta(state, oldName, newName)` — 内存中迁移 meta ✅
2. `scheduleSkillsStateSave()` — 500ms 防抖定时器，**不立即写盘** ❌
3. `refreshSkills()` — 立即从磁盘重载 `skillsState`，**覆盖内存迁移** ❌

**第二层（V2.12.1 CDP 验证发现的真实根因）**: 第一次修复将 `scheduleSkillsStateSave()` 替换为 `flushSkillsStateSave()`，但 `flushSkillsStateSave()` 内部含 `if (this.skillsStateSaveTimer === null) return` 提前返回。在 `openEditSkillDialog` 中 `renameSkillMeta` 后直接调用 `flushSkillsStateSave()`，此时 timer 为 null（因为之前没有 `scheduleSkillsStateSave()` 调用），导致 flush **不保存**，`refreshSkills` 重载旧 state 覆盖内存迁移。

时序图（V2.12.1 第一次修复后，仍有 bug）：
```
renameSkillMeta → flushSkillsStateSave() → refreshSkills(立即)
                      ↓                        ↓
                 timer===null → 提前返回      loadSkillsState 从磁盘读旧 state
                 （不写盘！）                    ↓
                                          内存迁移被覆盖，旧名 meta 残留
```

**最终修复**: 移除 `flushSkillsStateSave()` 的 `if (this.skillsStateSaveTimer === null) return` 提前返回，改为总是落盘（仅当 timer 不为 null 时清理 timer）。

**关键**: `renameSkillMeta` 函数本身正确（skillsState.ts:215-224），问题在调用时序 + flush 守卫。CDP 在真实 Obsidian runtime 验证才暴露第二层根因（代码级 Test 9 直接调 `saveSkillsState()` 绕过了 `flushSkillsStateSave` 的 timer 检查，未能发现此 bug）。
