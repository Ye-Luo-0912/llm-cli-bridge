# V2.14.0-C - On-Demand External Read Authorization Report

## GrantModel

V2.14.0-C replaces user-maintained external read roots with dynamic read grants.

Grant inputs:

- `sessionReadGrants`: paths approved during the current agent session after a pending external read request.
- `attachmentReadGrants`: paths explicitly provided by the user, such as files dragged into the composer.

Derived roots:

- `readRoots = vaultPath + sessionReadGrants + attachmentReadGrants`.
- `writeRoots = vaultPath + Vault-internal outputDir`.
- Grants are file-scoped by default, so approving one external file does not implicitly authorize sibling files or the whole directory.
- Directory-scoped grants remain possible through the `match: "directory"` field, but V2.14.0-C does not expose a UI for creating them.

No user setting page is added for external read roots. The user does not need to preconfigure external directories.

## DecisionFlow

Read decisions:

1. Vault-internal read is allowed by the default Vault read root.
2. Vault-external read without a grant returns `confirm` with reason `pending_read_request`.
3. If the user approves the pending request, the caller can add a session-level read grant.
4. A matching `sessionReadGrant` allows the same external file for the current session.
5. A matching `attachmentReadGrant` allows a user-provided attachment file.
6. Sensitive reads keep the existing sensitive policy: default `deny`, or `confirm` only when strong-confirm mode is explicitly enabled.

Write decisions:

- Vault-external write remains hard denied.
- Vault-external delete remains hard denied.
- Vault-external rename remains hard denied.
- Sensitive write/delete/rename remains hard denied regardless of read grant state or strong-confirm mode.

## UserFriction

This model reduces setup cost:

- Users do not maintain external root lists.
- The agent can ask for a specific external file on demand.
- User-approved files become session grants, not permanent broad access.
- User-attached files are naturally authorized as attachment grants.

The model deliberately avoids directory scanning and broad root authorization in this phase.

## Tests

The V2.14.0-B/C policy tests cover:

- Vault-internal read is allowed by default.
- External ungranted read returns `confirm` / `pending_read_request`.
- Approved session read grant allows the same file.
- Session grant does not allow sibling files by default.
- Attachment grant allows the attached file.
- External write/delete/rename are denied even when a read grant exists.
- Sensitive read/write behavior from B1 remains intact.
- `promptPackage`, AgentEvent v0.1, Skills runtime, CLI backend, and SDK backend are not wired to file access in this phase.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

- There is not yet a persisted session grant store. V2.14.0-C only defines the policy shape and behavior.
- There is not yet a pending-request UI. Future UI must show the exact requested path, operation, risk, and reason.
- Attachment grants are not yet wired to composer attachments.
- Directory grants should remain hidden until there is a dedicated review UX.
- No file content is read, scanned, or injected into prompts in this phase.

## Recommendation

Proceed next with a narrow storage/UI phase:

1. Add an in-memory session grant store.
2. Add a pending external read request queue.
3. Display exact-path confirmation UI for pending reads.
4. Wire composer attachments to `attachmentReadGrants`.
5. Continue to block external write/delete/rename and all sensitive mutations.
