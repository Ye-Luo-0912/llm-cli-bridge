## ProjectRuntimeConfigPatch

The plugin now resolves Claude runtime config in code instead of relying on `Start-Obsidian-Portable.cmd`:

- `src/claudeRuntimeConfig.ts` provides shared resolution and scoped env application.
- `src/claudeCliBackend.ts` applies project/local Claude config in `buildRunEnv()`.
- `src/sdkBackend.ts` applies the same config around SDK `query()` and restores the previous process env afterward.

Recommended optional project JSON:

```json
{
  "version": 1,
  "claudeConfigDir": "../LLM-AgentRuntime/private/claude-config"
}
```

The JSON stores only the Claude config path. Tokens remain in the local Claude settings and are not copied into plugin settings. `ANTHROPIC_CONFIG_DIR` is not used.

## RemainingRisk

- SDK env application uses scoped `process.env` mutation because the current SDK reads Claude config from environment. The code restores values after `query()` completes.
- Concurrent SDK runs could still share process-level env during overlap; this is acceptable for the current single-view smoke path but should be revisited if parallel SDK runs are introduced.
- The global user settings may still point at a failing provider; the plugin should prefer project/local config for the test Vault.

## Recommendation

Proceed with release-readiness validation using the project/local Claude runtime config path. Do not rely on `Start-Obsidian-Portable.cmd` as the source of truth for Claude CLI/SDK config selection.
