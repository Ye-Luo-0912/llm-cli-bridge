# V2.14.0-H - Native Attachments + FileRef Prompt Index + Read Tool Policy Integration Report

## NativeAttachmentEntry

Composer attachment entry now prefers native file selection.

- `+ File` opens a native file input with multi-select.
- Drag/drop on the composer accepts multiple files when the runtime exposes native file paths.
- `+ Path` remains as a fallback/debug path entry.
- All entries route through the same attachment FileRef flow.
- Attachment grants remain file-scope; adding an attachment does not authorize its sibling files or containing directory.

## WorkingSetUI

The Working Set remains compact and inline.

- Multiple attachments can be added in one action.
- Chips show `displayName`, `kind`, `status`, `source`, `pathKind`, and `fileType`.
- Chips mark bounded text snippets as `bounded`; image/PDF/binary/unknown or oversized files remain `refs-only`.
- Removing an attachment ref also removes its attachment read grant and bounded snippet.

## PromptFileRefIndex

`promptPackage` now includes a FileRef metadata index.

- The index contains only active authorized/user-added refs.
- Fields are limited to `displayName`, `path`, `kind`, `fileType`, and `status`.
- Pending, denied, and unauthorized external refs are excluded.
- The index never includes file content.
- Bounded text snippets remain the only file body content that can enter the prompt, and only for user-added small text/markdown/json attachments.

## ReadToolPolicyGate

`fileToolPolicy.ts` adds a minimal read-tool gate model.

- `read` and `stat` evaluate through `fileAccessPolicy`.
- Unauthorized external `read`/`stat` requests generate pending read requests.
- `list` and `search` are allowed only for Vault or authorized directory roots.
- `list` and `search` do not create pending requests for unauthorized external paths.
- External write/delete/rename remain hard denied through `fileAccessPolicy`.
- Sensitive paths still deny or require strong confirmation according to the existing read policy; sensitive write/delete/rename remain denied.

## ClaudeReadHandoff

Image, PDF, binary, and unknown attachments are exposed as refs only.

- CLI/Claude Code path: the prompt tells the agent to use Claude Code `Read` on the indexed path if it needs to inspect a refs-only file.
- SDK path: streaming image/file content is not implemented in this phase.
- Future SDK plan: add capability-gated image/file streaming after permission policy, model support, and size limits are finalized.

## Tests

Added coverage verifies:

- Native picker and fallback path UI wiring.
- Drag/drop UI wiring.
- Multi-attachment Working Set behavior.
- Attachment grant remains file-scope.
- FileRef metadata index includes only active authorized/user-added refs.
- Pending and denied external refs do not enter prompt.
- Small text bounded ingestion still works.
- Image/PDF/binary attachments are not read.
- Sensitive attachments are not read.
- `read`/`stat`/`list`/`search` policy gate behavior.
- External sibling remains unauthorized.
- External write remains denied.

Required validation:

- `npm run build`
- `npm run test:unit`
- `npm run test:process`

## RemainingRisk

- Native file path availability depends on Obsidian/Electron exposing `File.path`; `+ Path` remains as fallback.
- Drag/drop is implemented at composer level only, not as a full file manager.
- FileRef index is metadata-only; actual file tool execution remains future work.
- SDK image/file streaming is intentionally deferred.

## Recommendation

Freeze V2.14.0-H if build/unit/process pass and backend diffs remain limited to metadata/policy modules. Next phase can wire real read/list/stat/search callers, but they must call `evaluateFileToolPolicy` before reading anything.
